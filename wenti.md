在看完提示词和代码后，我发现了一个比“提示词优化”更根本的问题。先说实话，再给方案。

## 根本问题：OpenAI 兼容模式下，模型根本看不到视频
看代码 gemini_analyzer.py:L371-L373 ：

GRSAI 和臻臻是 OpenAI 兼容代理，视频 URL 被当作 纯文本 发过去。模型收到的就是一行字符串 https://www.youtube.com/shorts/MOgT6UtcCCQ 。它看不到视频画面、听不到声音、不知道时间轴—— 所有时间码全是幻觉出来的 。

对比 Gemini 原生 API gemini_analyzer.py:L403-L412 ：

Gemini 原生 API 的 fileUri 会让 Gemini 去 YouTube 真正拉取并观看视频。 这就是为什么剪辑效果差——当前走 GRSAI/臻臻 时，模型在盲猜。

## 优化方案：两阶段提示词 + 视频帧送进去
你提的思路完全正确——先描述视频内容（带时间轴），再根据描述确定剪辑片段。但要做成这件事，必须先解决“模型看不到视频”的问题。

### 代码层面需要的改动（不改文件，只说方案）
_call_openai_compatible_api 在传入 URL 时，改为：

1. 先用 yt-dlp 下载视频到临时目录
2. 用 ffmpeg 每隔 2-3 秒抽一帧（产生 20-50 张截图）
3. 按时间顺序把截图作为多张图片 base64 发给模型（OpenAI 多模态 vision 格式）
4. 两阶段提示词
这样模型就能真正“看”到视频的时间轴了。Gemini 原生 API 路径不需要改，它本来就能看。

### 两阶段提示词设计（核心） 阶段 1：视频时间轴描述 阶段 2：基于时间轴提取高光
## 为什么两阶段比一阶段好
当前一阶段 优化后两阶段 模型做的事 同时"看视频+找高光+打分+排序" 第一步专心描述，第二步专心筛选 时间码准确性 幻觉严重 基于截图时间戳，精准 可调试性 黑盒输出，不知道模型看到了什么 阶段 1 的文本可人工核对 提示词复杂度 3000+ 字规则，模型容易丢失重点 阶段 1 无规则，阶段 2 规则简洁 跨供应商兼容性 不同模型对复杂指令遵循度不同 两阶段各自独立，容错高

## 总结
1. 根本原因 ：GRSAI/臻臻是 OpenAI 兼容代理，视频 URL 做不了视频分析，所有时间码都是幻觉
2. 解决路径 ：需要改 _call_openai_compatible_api 让它下载视频→抽帧→把截图发给模型，再加上两阶段提示词
3. 提示词本身 ：上面设计的两阶段结构已经就绪，核心思路就是你说的"先描述再剪辑"
